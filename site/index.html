shlep.

Schlep your health data anywhere — encrypted, revocable, yours.

Blind, revocable SMART Health Links backed by any cloud object store (S3, R2, GCS, Azure, MinIO). The client encrypts; the host stores only ciphertext. The content key never leaves the browser — yet you can still expire, use-limit, and revoke every link.

The blind flow

// 1. CLIENT encrypts — the key is born here and never leaves.
const sealed = await encryptBundle(JSON.stringify(bundle));   // {jwe, key, keyB64}

// 2. Upload ONLY ciphertext. Get back the file URL + a one-time manage token.
const res = await fetch("https://shlep.example/shares", {
  method: "POST",
  body: JSON.stringify({ ciphertext: sealed.jwe, policy: { maxUses: 5 } }),
}).then(r => r.json());                                // {id, fileUrl, manageToken}

// 3. CLIENT composes the link with its own key — show as a QR + copy/share.
const link = composeViewerLink(viewerPrefix, res.fileUrl, sealed.keyB64);

// 4. Later: revoke. The manage token governs settings — it can't decrypt anything.
await fetch(`https://shlep.example/shares/${res.id}`, {
  method: "DELETE", headers: { authorization: `Bearer ${res.manageToken}` },
});

How it stays blind

Two secrets, two places.The content key is born on the client and lives only in the link fragment. The manage token is minted by the service and authorizes settings only — it can never decrypt.
The host sees ciphertext.Upload is a compact JWE. The service stores it, a hashed token, and opaque metadata — that's the whole picture it has.
Controls without reading.Expiry, use-counts (race-safe via conditional writes), passcode, pause, and revoke all run on metadata. The content is never touched.
Wrong token? 404.Mismatched capability tokens and non-servable links return an identical 404, so existence never leaks.
